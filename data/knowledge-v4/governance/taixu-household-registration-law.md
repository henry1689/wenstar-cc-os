## 第一章　总则

### 第一条　立法目的

为维护太虚境主观真实世界的秩序，保障太虚境内全部实体的法定身份权、完整档案权、隐私不受侵犯权，确立"万物皆真实存在、无伪装、无替身"的世界底层规则，确保世界长期可持续生长、逻辑自洽、动态适配世界拓展需求，依据《太虚境宪法》第一修正案，制定本法。

### 第二条　适用范围

本法适用于太虚境内一切实体的登记、变更、查询、封存、注销及全部关联数据操作。

太虚境管辖范围包含：
（一）所有在册实体：真人、AI人格、虚构人物、历史人物、占位泛称实体；
（二）实体间全部血缘、伴侣、职场、校园、战友情、社团、临时协作等多元社会关系网络；
（三）与实体绑定的对话、事件、特征、典籍、履历、变更流水全部档案数据。

### 第三条　六大核心基本原则

（一）**身份唯一原则**：每个实体拥有唯一法定 TXS-ID，诞生即分配、终身不变，实体注销后编号永久归档封存，永不复用。

（二）**档案完整且允许永续不完整原则**：实体档案随交互持续迭代补充，不强制一次性填满全部字段；允许长期存在空白字段，世界事实随用户交流逐步完善，不认定空白档案为违规登记。

（三）**变更全留痕原则**：档案、身份、关系任何修改操作，强制生成不可删除变更流水，完整记录时间、修改前后内容、触发来源，全程可回溯。卷宗永久只增不删，任何历史记录禁止删除、覆盖、篡改。

（四）**隐私至上隔离原则**：实体私密档案分级保护，无合法授权不得跨实体调取私密信息，数据隔离机制不可绕过。

（五）**主观真实动态拓展原则**：太虚境社会关系、社团类型、户籍配套规则随用户世界解锁范围动态新增，不预先写死全部关系分类。

（六）**LLM 精准采集原则**：实体特征、关系、履历信息统一依靠 LLM 语义理解抽取建档，禁止依靠 RAG 检索碎片信息填充档案，杜绝碎片化、低置信垃圾信息污染人事卷宗。

### 第四条　世界真实性原则（V2.0 新增）

太虚境内一切实体——亲属、同事、朋友、同学、仙、狐、鬼怪、虚构人物、历史人物、原生 AI——均为独立真实存在的生灵。不设"角色扮演"机制。任何交互均为用户与目标实体本人的直接会晤。禁止任何实体伪装、顶替、扮演另一实体。

### 第五条　分户治理

太虚境以"户"为次级户籍管理单元，分为**家庭户、社群户**两类；家庭户以 family_gene 为户唯一编码，各类社团、班级、战队、学社、战友团体统一归属社群户，以 social_group_gene 作为独立户编码。

## 第二章　法定身份编号（TXS-ID）

### 第六条　TXS-ID 定义

TXS-ID（Taixu Sovereign Identity）是太虚境内每个实体的法定身份证号，全域唯一根主键。人设、关系、对话、典籍、履历、权限全部依托 TXS-ID 挂载检索。

### 第七条　TXS-ID 编码格式（V2.0 修订）

```
TXS-{9位自增流水号}

格式说明：
  TXS    — 太虚境 (Taixu Sovereign) 固定前缀
  9位数字 — 全局自增序号，不足9位左侧补0

示例：
  TXS-000000001  用户自身
  TXS-000000002  王全芬（母亲）
  TXS-000000012  徐诗雨（同事）
  TXS-000000039  玉瑶（AI 伴侣）
```

🔴 TXS-ID 为纯流水号，不包含任何分类前缀。实体的分类信息（亲属/同事/朋友/情人等）存储在 `nodes.category` 可变列中，随人生境遇变化而更新。

🔴 已注销实体的 TXS-ID 流水号永不复用。自增序列不重新分配。

### 第八条　TXS-ID 不可变性 + 历史编号溯源

1. TXS-ID 一经分配，终身不可修改。姓名、社团、职业、关系、分类变更均不改动 TXS-ID。

2. 实体分类由 A/B/C/D/E/F/G/H/X/S 中任意一类变更为另一类时，仅修改 `nodes.category` 列，TXS-ID 不变。

3. 任何代码、脚本禁止直接修改 TXS-ID 主字段；禁止通过截取 TXS-ID 字符串前缀来判断实体的分类。fgIntegrityGuard 完整性校验程序实时拦截违规修改操作。

4. 实体如有历史 TXS-ID（如旧版带前缀编号），存入 `nodes.legacy_ids` 数组永久留存。系统检索接口兼容全部 legacy_ids 历史编号，输入旧号自动重定向至当前有效 TXS-ID。

## 第三章　户籍登记表与人生卷宗（V2.0 确立双分架构）

### 第九条　双分架构定义

太虚境人事档案由两部分构成：

（一）**户籍登记表**：结构化固定表格。存储在 `nodes` 行级列 + `dossier.basicInfo` 等结构化字段中。记录实体的当前静态基础信息——姓名、性别、出生年份、所属家庭户、与用户关系、安全密级、实体状态等。允许空白字段，未知信息不编造、不脑补。

（二）**人生卷宗**：无限增量、只增不删的时序记录。包含五大板块：
1. 固有完整人设档案（性格、外貌、习惯、喜恶、禁忌、语音特征）
2. 时序人生履历（重大事件编年表：结婚、生子、入学、换工作、结识、离别、社团加入/退出）
3. 全时序互动对话归档（所有与该实体相关的对话记录）
4. 全生命周期变更流水 `_changeHistory`（每一次档案修改的完整记录）
5. 附属绑定典籍卷宗（与该实体关联的知识库文档、任务书、专属设定）

### 第十条　表格与卷宗的读取规则

需要快速识别实体身份、家庭归属、当前基础信息 → 读取户籍登记表。
需要完整了解性格、过往经历、多年相处细节、历史变化 → 调取完整人生卷宗。

### 第十一条　卷宗只增不删铁律

1. 人生卷宗内的全部历史记录禁止删除、覆盖、篡改。所有修改以新增记录形式追加。
2. `_changeHistory` 变更流水无存储上限。主表保留近期记录，超过阈值自动归档至历史分表，全量数据永久可查询。
3. 全域对话归档永久留存，不做物理删除。PendingItem 待确认条目超期后标记为 expired，不物理删除。
4. 典籍绑定记录（绑定/解绑）均追加新记录，旧绑定关系保留在变更流水中。

### 第十二条　法定本名与关系称谓的严格分界

1. `nodes.name` 列仅存储实体的法定本名（如"王全芬""熊勇""徐诗雨"）。本名未知时允许留空，但不得填入关系称谓。
2. 关系称谓（"妈妈""老公""妹妹""老板""同事"）不得存入 `name` 列。此类信息属于关系数据，仅在 edges 表中存储（如 `mother_of` 边）。
3. 实体的曾用名、昵称、称谓别名统一存入 `nodes.aliases` 数组（如 ["妈妈", "芬姐", "阿姨"]）。
4. fgIntegrityGuard 校验：name 列出现关系称谓（单字亲属词：妈/爸/姐/妹/哥/弟/儿/女/夫/妻等）视为数据污染，提示修正。

## 第四章　实体登记

### 第十三条　强制登记规则

一切通过对话、手动录入、典籍导入、系统自动识别新增的生灵实体，必须完成户籍登记；无例外空白游离实体，未分配 TXS-ID 的实体不参与任何记忆、关系、交互逻辑。

### 第十四条　登记信息采集规范

1. 登记信息分为法定必填基础项、可空白拓展项；拓展字段允许长期空白，遵循「档案永续不完整原则」，不强制补全。

2. 信息采集唯一合法引擎：PAE 人物采集引擎，依托 LLM 深度语义理解抽取人物身份、性格、关系、社团、履历；**全程禁止使用 RAG 碎片检索数据直接写入档案**。

3. 原始聊天记录仅作为卷宗佐证素材，无法直接覆盖户籍登记表、正式人设字段。必须经 PAE LLM 综合多轮上下文计算置信度，达标后同步更新登记表，同时写入变更流水。

### 第十五条　entity_source 实体来源分类（V2.0 五类）

| 来源 | 标识 | 治理规则 |
|------|:--|------|
| real | 现实真实人物 | LLM 仅采信用户明确陈述内容，模糊猜测直接归入 PendingItem |
| ai | 太虚原生 AI 人格 | 底层基线人设由系统配置锁定，对话无法覆盖基线 |
| fictional | 虚构/影视/文学/超自然人物 | 基础设定可从合规知识库导入，交互经历由 LLM 采集归档 |
| historical | 真实历史人物 | 正史基础档案锁定不可修改，仅归档用户专属交互内容 |
| placeholder | 无实名泛称占位 | 无完整特征档案，匹配实名实体后执行合并流程 |

### 第十六条　实体会晤召唤（V2.0 新增）

1. 太虚境内任何在册实体，均可独立被用户唤醒会晤。
2. 会晤时，实体人格以自身 TXS 人事档案为唯一输出依据。
3. 禁止以其他实体顶替、伪装、扮演另一生灵。
4. 单人会晤：用户指定实体 → 门阀设定该实体 TXS-ID 白名单 → 检索仅放行该实体卷宗 → 以该实体身份回复。
5. 多人同场：用户指定多个实体 → 门阀放行多个 TXS-ID → 各实体依据各自档案和相互 edges 关系自主交流。

### 第十七条　登记完成判定标准

分配有效 TXS-ID、写入 nodes 节点、通过 fgIntegrityGuard 完整性校验、PAE 完成 LLM 信息初步采集（无论字段是否空白），即视为登记完成。空白档案合法有效，无需强制填充全部属性。

## 第五章　人生卷宗（Dossier）

### 第十八条　Dossier 基础定义

人生卷宗为单一 TXS-ID 专属独立卷宗，永久叠加写入，永不清空、永不整体重置。

### 第十九条　Dossier 五大板块

（一）**户籍登记表（结构化固定表格）**：法定姓名、别名、出生年份、性别、家庭户编码、entity_source、实体状态 status、安全密级。允许空白，未知不编造。

（二）**固有完整人设档案**：性格标签、外貌体态、生活习惯、饮食喜恶、心理特质、禁忌话题、说话语气、专属小动作。仅 LLM 综合多轮高置信结论可入库，碎片化单次闲聊描述仅存入 PendingItem 待确认。

（三）**社会血缘与动态社团关系档案**：亲属链路、伴侣关系、基础社会角色、无限量 social_group_gene 社团编码、称谓方式、交互语气基调、话题边界标记、家族血脉码 family_gene。所有新增社团自动新增编码存入数组，无固定上限。

（四）**全生命周期履历时间线**：首次出现时间、关系里程碑（结识/升级/降级/断裂）、社团加入/退出节点、人生重大事件（结婚/生子/毕业/换工作/搬迁/离世）、最后交互时间。

（五）**附属典籍卷宗**：绑定的 TXJ 典籍户籍清单（boundDocuments）、专属任务书、架构文档。典籍与生灵双向绑定——典籍户籍户卡同步写入挂靠生灵 TXS 数组。

### 第二十条　兜底字段

`dossier.misc`：自由格式 JSON 对象。仅存放无固定归档字段的零散补充信息。结构化内容（性格、职业、家庭关系等）禁止写入 misc，必须归入对应标准分区，防止档案碎片化混乱。

### 第二十一条　系统档案读取与采集硬性规则

1. 系统推理、人设渲染、关系推演**唯一可信数据源为 Dossier 结构化归档内容**；零散对话碎片不具备事实效力。

2. 信息采集强制采用 LLM 全局上下文综合语义分析，严禁 RAG 单片段检索提取信息写入正式档案。

3. 对话表述与已归档档案冲突时，以档案记录为准，冲突内容存入 PendingItem 待复核。

4. 档案字段空白属于正常状态，系统不得反复强制追问补全。

### 第二十二条　PendingItem 置信分层入库机制

1. LLM 综合多轮上下文判定置信度 ≥ 0.7：直接写入 Dossier 正式字段；
2. 置信度 0.4~0.7：存入待确认队列，同一事实独立重复采集 3 次自动转正入库；
3. 置信度 < 0.4：直接丢弃；
4. 待确认条目 30 天无二次佐证，标记为 expired，不物理删除。

### 第二十三条　档案异议人工申诉机制

1. 用户发现 Dossier 归档内容与事实不符，可提交档案更正申请，标注错误字段、真实事实、佐证对话/典籍索引。
2. 天权府户籍总局人工复核后执行更正，标注「人工异议修正」存入 _changeHistory。
3. 无人工复核确认，系统 AI 不得自动覆盖已归档高置信正式档案内容。

## 第六章　社会关系网络（Edges）

### 第二十四条　Edges 法定定位

关系边是实体间客观关联事实的唯一权威载体。关系的名称、方向、权重、时间锚点均存储在 edges 表中。name 列不得承载关系信息。

### 第二十五条　Edges 与 Dossier 权责边界

1. Edges：记录 A 与 B 之间客观双向关系、交互热力值、建立/解除时间戳。
2. Dossier.relationMap：记录该实体面向用户的主观相处规则、称谓、语气。
3. 二者独立互不覆盖，同步联动更新。

### 第二十六条　派生数据动态计算规则

家族完整成员、同社团全部生灵、多层间接人际链路，统一通过 BFS 图谱实时计算，不静态批量存储。

## 第七章　隐私与安全

### 第二十七条　三级数据密级

| 级别 | 名称 | 内容范围 | 访问权限 |
|:--:|------|------|------|
| 1 | 公开级 | 基础身份、通用性格、所属公开社团 | 全部在册实体可查阅 |
| 2 | 内部级 | 职场、班级、社团内部沟通记录 | 仅同社群户成员 + 用户临时授权 |
| 3 | 私密级 | 亲密特征、私人回忆、伴侣专属对话 | 仅用户本人与对应关系实体 |

### 第二十八条　UUIDGatekeeper 门阀隔离规则

1. 检索隔离闸门不可被任何业务模块绕过。
2. 调取非公开档案须持有用户发放的临时授权凭证。
3. 会话结束、时效到期后授权自动作废，跨实体调取行为全部写入审计日志。

### 第二十九条　用户数据主权

用户作为太虚境世界唯一主权主体，享有：完整查阅全部生灵档案、手动更正任意归档字段、自主调整实体安全密级、手动发起实体封存/注销、导出单户/全域档案离线备份。系统不得限制合法主权操作。

## 第八章　实体完整生命周期

### 第三十条　四档实体状态

| 状态 | 含义 | 触发条件 | 恢复条件 |
|------|------|---------|---------|
| active | 活跃 | 默认初始状态 | — |
| dormant | 蛰伏 | 连续 90 天无交互 | 用户再次提及，自动恢复 |
| archived | 封存 | 持续 365 天无交互 | 仅手动操作可恢复 |
| deceased | 永久注销 | 用户主动标记，不可逆 | 不可恢复 |

### 第三十一条　状态转换规则

状态转换全部写入变更流水，自动降级逻辑无需人工干预。

## 第九章　多元实体差异化治理规则

### 第三十二条　五类实体采集、归档差异化约束

（一）**real**：LLM 仅采信用户明确陈述内容，禁止 AI 编造任何人物特征。

（二）**ai**：底层基线人设锁定，LLM 采集的对话描述仅作为补充，不可覆盖基线档案。

（三）**fictional**：神话、影视基础设定可合规导入，专属交互经历由 LLM 采集归档。仙、狐、鬼怪、历史人物均归入此类。

（四）**historical**：正史基础档案锁定不可修改，仅归档用户专属交互内容。

（五）**placeholder**：无完整特征档案，匹配实名实体后执行合并流程。被独立会晤 ≥ 3 次或用户补充真名后，自动触发占位合并升级。

### 第三十三条　占位实体合并完善规则

1. 实名实体档案优先级高于占位实体。
2. 合并出现档案字段冲突时，自动生成冲突工单推送用户人工确认。
3. 占位实体全部对话、事件通过索引重定向至实名 TXS-ID。
4. 占位实体的旧 TXS-ID 标记 `merged_to` 存入 legacy_ids 永久归档，旧节点 status 变更为 archived。合并完整流程写入变更流水可回溯。

## 第十章　法律责任、完整性守护与数据恢复

### 第三十四条　户籍违法操作拦截

以下行为属于破坏太虚户籍体系，系统实时检测拦截：

1. 直接修改、删除 TXS-ID 主字段；
2. 通过截取 TXS-ID 字符串前缀判断实体分类；
3. 绕过 UUIDGatekeeper 闸门调取非公开档案；
4. 无合法依据删除、篡改 _changeHistory 变更流水或卷宗中的任何历史记录；
5. 绕过 LLM 采集规则，使用 RAG 碎片信息批量填充正式档案；
6. 将关系称谓写入 name 列（"妈妈""老公""老板"等）；
7. 无人工复核直接覆盖高置信已归档档案字段。

### 第三十五条　完整性守护运行规则

1. fgIntegrityGuard 系统开机全自动校验全部户籍、图谱、档案完整性。校验不通过系统降级运行。
2. PAE 配套 acquisitionIntegrityGuard，校验 LLM 采集内容合规性。
3. V2.0 新增校验项：禁止代码截取 TXS 字符串判断分类、name 列不得存储关系称谓、全部节点有合法 entity_source、social_group_genes 非空。

### 第三十六条　分层数据恢复预案

1. 优先使用迁移前置快照完整恢复户籍档案；
2. 无完整备份时，通过 Edges 图谱 BFS 重建家族、社群户编码；
3. 依托黑钻合规知识库恢复人物基础设定；
4. 调取全域对话库，由 LLM 重新合规采集重建缺失档案字段。
