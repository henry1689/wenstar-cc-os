# /no-patching — 禁止打补丁原则

## 触发条件

任何时候发现新场景暴露出了现有代码无法覆盖的问题。

## 核心原则

每遇到一个新场景暴露出的问题，不是往现有代码加一个 if，而是改设计让这个 if 变得不必要。

## 判定标准

| 补丁行为 ❌ | 架构修复 ✅ |
|---------|---------|
| `if (这个场景) 注入这个` | 改为数据驱动，无条件分支 |
| 在现有函数中加一个 else 分支 | 重新设计函数使其不需要条件判断 |
| 复制一段代码改几个参数 | 提取参数化函数 |
| 在调用链中间加一层检查 | 重新设计调用链使其自然覆盖 |

## 自检清单

在每次修改前问自己三个问题：

### 问题 1：这个修改是补丁还是架构修复？

```
如果我的修改加了条件判断（if/switch/三元）→ 补丁 ❌
如果我的修改改了数据结构或流程 → 架构修复 ✅
```

### 问题 2：未来出现类似场景是否需要再改？

```
如果来一个"问家人年龄"我又得加一个 if → 架构没改对 ❌
如果"问任何人年龄"都自动覆盖了 → 架构对了 ✅
```

### 问题 3：我是在让代码更简单还是更复杂？

```
加了一个 if → 代码更复杂了 ❌
去掉了一个 if → 代码更简单了 ✅
```

## 整改流程

```
发现新场景 → 识别这个场景在架构中的位置
         → 现有设计是否能自然覆盖？
         → 能 → 无需修改
         → 不能 → 重新设计那一层，使其从结构上覆盖这个场景 + 未来的同类场景
               → 验证新设计不需要为这个场景加特定 if
```

## 验收标准

架构修改完成后，以下条件必须全部满足：

- [ ] 新场景自然覆盖（无需额外条件判断）
- [ ] 至少一个类似场景也自动覆盖（证明非特例）
- [ ] 代码行数减少或持平（没有因为"考虑未来场景"而膨胀）
- [ ] 没有新增条件分支（if/switch/三元）
