### 7. 🔴 Bug 修复强制流程：六步闸门

**核心原则**：调试不是补洞，而是维护系统一致性。不允许在未经分析时修改代码。

```
闸门1 · 诊断（禁止改代码）
  ├─ 复现路径：输入 → 期望 → 实际
  ├─ 根因（5 Why）：不是失败在哪一行，是为什么失败
  ├─ Bug 分类：实现错误 | 数据模型不一致 | 状态管理混乱 | API契约破坏 | 架构边界破坏 | 需求歧义
  └─ 输出调用链与数据流：数据从哪来 → 谁拥有 → 谁可改 → 生命周期
     ↓
闸门2 · 影响范围（禁止改代码）
  ├─ 列出会被影响的文件、函数、接口、测试
  └─ 是否存在同类问题散落在其他地方？（只记录，不顺手改）
     ↓
闸门3 · 方案选择（禁止改代码）
  ├─ 至少两个方案 + 利弊 + 推荐
  ├─ 确认不违反系统不变量（见下方）
  └─ 等用户批准
     ↓
闸门4 · 限定实施（允许改代码）
  ├─ 改动预算：bugfix ≤3个文件, ≤100行; 涉及新状态/新API/新字段 → 先讨论
  └─ 不顺便重构、不顺手清理、不"我还发现另一个问题也修了"
     ↓
闸门5 · 回归验证
  ├─ 编译 + 原有测试 + 新增测试
  └─ 回归清单：主流程 / 原bug场景 / 空值边界 / 权限错误 / 公共API未变
     ↓
闸门6 · 架构自检 + 变更报告
  ├─ 是否保持了模块边界？是否引入重复逻辑？是否加了特殊分支？
  └─ 改善报告（根因·改了什么·核验证据·注意事项）→ 等确认 → git commit
```

**铁律**：闸门 1-3 阶段绝对禁止改代码。违反任何一次直接停止。

任何 `.ts`/`.js` 文件被修改后，按以下流程走完：

```
修改代码
  ↓
① 五道机械核验（编译·功能·牵连·服务·监控）
  ↓  有一道不过 → 回退→诊断→修→重验
  ↓  五道全过 ↓
② 第六道行为核验 ★必做★
  发测试消息验证 LLM 回复不出现以下问题：
  - 角色称呼泄漏(鸿艺叔/梓铭/爸爸/叔叔)
  - 身份混淆(说自己是诗韵/诗雨等角色)
  - 场景循环(浴缸等旧场景)
  - 重复回复("你再说一遍好不好")  
  ↓  行为不过 → 回退→诊断→修→重验
  ↓  行为通过 ↓
③ 提交（本地 git commit）
  ↓
④ 输出改善报告（根因·改了什么·核验证据·注意事项）
  ↓
⑤ 问用户："是否上传同步？"
  ↓
⑥ 用户确认后才 git push
```

**铁律**：
- 永远不自动推送。每次推送前必须等用户确认。
- 改善报告必须有核查证据，不是"应该没问题"。
- 行为核验只测机械核验过不了的——**编译通过≠聊天正常**。
  
判定标准：
| 补丁行为 | 架构修复 |
|---------|---------|
| `if (这个场景) 注入这个` | 改为数据驱动，无条件分支 |
| 在现有函数中加一个 else 分支 | 重新设计函数使其不需要条件判断 |
| 复制一段代码改几个参数 | 提取参数化函数 |
| 在调用链中间加一层检查 | 重新设计调用链使其自然覆盖 |

**铁律**：每遇到一个新场景暴露出的问题，不是往现有代码加一个 if，而是改设计让这个 if 变得不必要。

### 12. 🔴 系统性变更后 → 强制复审（2026-07-24 血泪经验）

**背景**：本轮 UUID 贯穿全链路改善涉及 6 个文件、3 个数据库表、2 条 SQL，通过了编译、启动、5 实体测试——全部"通过"。但在复审中发现：

- `this.save()` 缺失 → sql.js 回填数据全部在内存中，重启即丢失
- 标注率从 40.7% 降到 0%——**验证通过了但数据没有持久化**

**教训**：验证 ≠ 复审。验证测的是"当前是否工作"，复审查的是"改了之后会不会坏"。

**复审流程**（每次 >=3 个文件或涉及数据库变更后强制执行）：

```
第 1 遍 · 对比审计
  ├─ 逐条对比计划 vs 实际：计划中每一行改动是否都完成了？
  ├─ 列出的每条铁律是否都通过了硬数据验证？
  └─ 有没有"顺手多改的"或"忘记改的"？

第 2 遍 · 持久化审计（涉及数据写入时）
  ├─ 停止服务 → 重启服务 → 再次查询数据库
  ├─ 写入的数据是否还在？（标注率、新增列、回填结果）
  └─ 铁律：只要数据没写入磁盘，修复就没完成

第 3 遍 · 边界审计
  ├─ 正常路径（用户实际会用到的场景）
  ├─ 会晤路径（每个实体 ≥2 个不同话题测试）
  ├─ 异常路径（API 降级、空数据、会中说"瑶瑶"退出）
  └─ 铁律：3 类路径都通过才算复审完成
```

**复审时间要求**：至少和修改本身花同样多的时间。修改 30 分钟 = 复审至少 30 分钟。

**复审发现率记录**：
- 2026-07-24 UUID 全链路改善：5 次验证全部通过，复审发现 1 个致命问题（`save()` 缺失）
- 经验：大型改善每次都会有一次验证查不出的问题——复审是最后防线

## 🔴 实体会晤反编造体系（2026-07-22 根因分析沉淀）

### 问题模式

用户问实体角色"记不记得我们之前……"，实体编造了不存在的往事。
**根因不是 LLM 不听话，而是系统没有告诉它"记忆的边界在哪里"。**

### 三层根因（5 Why）

| 层 | 根因 | 为什么致命 |
|---|------|----------|
| 1 | **EntityContextBuilder 规则只约束"档案"，不约束"记忆"** | 规则说"档案没写的你不知道"，但"过去发生的事"在【过去的对话记忆】里，不在"档案"里。LLM 认为"我可以基于对话历史自由发挥" |
| 2 | **MemoryInjector 剥离了所有结构标签** | `replace(/【[^】]*】/g, '')` 把【我的档案】【过去的对话记忆】【我的家人/熟人】全部变成 💭 前缀，LLM 分不清事实 vs 片段 vs 关系——全部混在一起，自然会"借题发挥" |
| 3 | **userMessage 被 KB 文本污染** | 会晤模式下 knowledgeBaseText 没被隔离墙清除，拼到 userMessage 前面送入 LLM，大量的 KB 内容让 LLM 觉得"我知道很多，可以说很多" |

### 五层防护（已实施）

```
L1 · 系统提示记忆边界
    EntityContextBuilder: "【过去的对话记忆】中的片段就是你对过去的全部记忆"
    
L2 · 稀疏记忆哨兵
    MeetingContextPipeline: 检索结果 ≤2 条 → 追加【⚠️ 记忆边界】强制规则
    
L3 · 保留结构标签
    MemoryInjector.preserveLabels=true → 会晤模式保留【我的档案】【过去的对话记忆】
    
L4 · 隔离 userMessage
    chat.ts: 会晤模式 _meetingEntityName ? message : 正常拼接
    
L5 · 事后编造检测
    FabGuard: 每轮回复检查回忆性断言是否有 memoryFragments 支撑
```

### 涉及文件

| 文件 | 改了什么 |
|------|---------|
| [EntityContextBuilder.ts](src/m4/household/EntityContextBuilder.ts) | +2 条记忆边界铁律 |
| [MemoryInjector.ts](src/m4/MemoryInjector.ts) | +preserveLabels 参数，会晤模式保留结构标签 |
| [chat.ts](src/webui/chat.ts) | preserveLabels=true, userMessage 隔离, FabGuard 检测 |
| [MeetingContextPipeline.ts](src/webui/chat/MeetingContextPipeline.ts) | 记忆 ≤2 条注入哨兵, 零记忆注入禁止规则 |

### 推而广之的方法论

> **任何"让 LLM 少说"的需求，不应该通过改 prompt 的措辞来实现——应该通过"告诉它信息边界在哪里"来实现。**
>
> LLM 编造不是因为"它不听话"，而是因为"它不知道自己不知道什么"。
> 解决编造的正确姿势：**明确列出"你知道的全部"，然后说"除此之外你不知道"。**

## 🔴 会晤记忆检索与回忆叙事体系（2026-07-23 根因分析沉淀）

### 问题模式

用户问实体角色"诗韵，你还记得我们之前……"，实体说不记得或不自然地堆砌碎片。

**根因不是数据没有，而是 LLM 拿到的是碎片列表（"鸿艺说:…|你说:…"），加上反编造规则压死——"没精确匹配就不准说"。**

### 五层根因 + 修复

| 层 | 根因 | 修复 |
|---|------|------|
| 1 | **家族关系检测用正则匹配标签文字**——`/姐姐|妹妹/.test(relationLabel)` 漏掉 `younger_sister_of` | EntityContextBuilder: 改用 `SIBLING_RELS = new Set(['elder_sister_of','younger_sister_of',...])` 精确匹配 relation type |
| 2 | **家人和社交混在一起**——acquaintance_of 和 sister_of 全堆在同一个`【家人/熟人】`区块 | MeetingContextPipeline: 分离`【我的家人】`和`【我认识的人】`，家族边用反向标签（"她是我的姐姐"而非"我是她的妹妹"） |
| 3 | **`relation_to_user` 被 FG 启动迁移反复覆盖**——写入正确值后重启又恢复 | EntityContextBuilder: 内联 `KNOWN_FIXES` 硬编码已知正确值（不依赖数据库字段），在展示覆盖迁移 |
| 4 | **记忆以碎片列表注入而非叙事格式**——LLM 看到的是 15 条 `鸿艺说:…|你说:…` | MeetingContextPipeline: 增加回忆规则——"用你自己的话组织叙述…像朋友说'哎对了那次我记得…'" |
| 5 | **泛化问题（"我们以前聊过什么"）ngram 排序全部等分，等于随机** | MeetingContextPipeline: 泛化问题改用时间均匀采样（头-中-尾三段各取等量） |

### 全系统适用原则

> **回忆 ≠ 编造**。记忆是真实发生过的事，LLM 应该像真人一样自然讲述。
>
> **碎片列表 ≠ 记忆**。真人不会报菜名式地背诵聊天记录——LLM 也不应该。
>
> **关系标签要从 edges 计算**，不能依赖 FG `relation_to_user` 字段（被迁移反复覆盖）。
>
> **家族边用 relation type 判断**（`SIBLING_RELS`），不用正则匹配标签文字。
