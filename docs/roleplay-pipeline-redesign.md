# 🏗️ 角色扮演域 —— 完整架构设计方案 v2

> 合并版本：原五步管线方案 + 独立业务域 + 三阶生长 + 三库对齐 + 全链路监管 + 离线自巩固

---

## 一、现状与痛点

角色扮演代码散布在 `chat.ts` 的 2500 行中，经历了十几轮修补，形成了"修一个漏一个"的局面。

### 当前工作流

```
用户说"你多大了"
  ↓
M1 DNA编码 → LLM实体提取 → 提取到"姐姐"但没提取到"年龄"
  ↓
M4 家族图谱 → FG分支加载 → 诗韵的画像无 age 字段
  ↓
知识库检索 → 搜索"年龄" → 0 结果
  ↓
上下文扫描 → 历史中没有"诗韵X岁" → 无结果
  ↓
knowledgeBaseText 碎片化拼接 ← 7个步骤追加/覆盖，顺序混乱
  ↓
LLM 生成 ← 没数据 + 没强制"不知道"指令 → 编造"二十一"
  ↓
RoleplayHealthGuard 检查 → 只能打日志，没有实际约束力
```

### 七个根本病因

| # | 问题 | 描述 | 根因 |
|:-:|------|------|------|
| 1 | **代码散布在 chat.ts** | 角色扮演逻辑混杂在 2500 行主对话代码中 | 没有独立业务域 |
| 2 | **数据采集与注入分离** | 查FG、查KB、注入各是一处，互不知道对方结果 | 没有统一的数据收集器 |
| 3 | **覆盖逻辑不可预测** | knowledgeBaseText 被 7 个步骤追加/覆盖 | 没有确定性注入顺序 |
| 4 | **没有数据就绪确认** | 不管查到没有，直接发给 LLM | 缺少"就绪门" |
| 5 | **反编造是事后补救** | LLM 先编完，再检查 | 应该是事前预防 |
| 6 | **监管层没有执行权** | HealthGuard 只能打日志，不能阻止编造 | 没有回溯修改机制 |
| 7 | **角色记忆不沉淀** | 聊完就忘，没有三库流转 | 没有角色专属记忆体系 |

## 二、架构定位：独立业务域

角色扮演从 `chat.ts` 完全剥离，升级为和 **记忆域、关系域、感知域** 平级的独立业务域。

### 架构边界

```
src/app/roleplay/                      ← 所有角色扮演逻辑收敛至此
├── RoleplayDomain.ts                  ← 域入口（chat.ts 唯一调用点）
├── DataCollector.ts                   ← ① 全量数据采集
├── ReadinessGate.ts                   ← ② 数据就绪判定
├── PromptAssembler.ts                 ← ③ 确定性提示词装配
├── Validator.ts                       ← ⑤ LLM 回复验证
├── RoleplayProfileManager.ts          ← 三阶生长：角色档案管理
├── RoleplayMemorySync.ts             ← 三库记忆同步
├── RoleplayAuditor.ts                 ← 审计日志 + 指标上报
├── RoleplayOfflineTask.ts             ← 离线巩固任务
├── RoleplayPromptBuilder.ts           ← [已有] 规则①-⑥构建器
├── CharacterProfileScanner.ts         ← [已有] 上下文扫描
├── EmotionSnapshot.ts                 ← [已有] 情感快照
├── RoleplayHealthGuard.ts             ← [已有] → 简化为 Validator 入口
└── types.ts                           ← 域内类型定义
```

### 接口化交互

```
角色域 ──→ MemoryOrchestrator  ← 记忆读写（三库对齐）
角色域 ──→ FamilyGraph          ← 图谱查询（FG分支隔离）
角色域 ──→ LLM Provider         ← 模型调用（复用主系统）
角色域 ──→ KnowledgeBase        ← 知识检索
```

角色域**不直接操作底层数据库**，所有读写走主系统标准接口，天然遵守主系统所有架构规则。

### 数据标记化

所有角色扮演数据统一带双标记：

| 标记 | 值 | 用途 |
|------|-----|------|
| `roleplay_id` | `rp_{角色名}_{sessionId}` | 按角色召回/隔离 |
| `source` | `roleplay` | 按来源统计/清理 |

同库同表但逻辑隔离，召回、统计、清理都能精准区分，不污染主人格。

## 三、核心管线：五步闭环

```
用户输入 → ① DataCollector → ② ReadinessGate → ③ PromptAssembler
                                                 → ④ LLM Generate
                                                 → ⑤ Validator
                                        ↑失败→ 回退重试 | 告警
```

### 第一步：DataCollector（数据采集器）

**职责**：单次、并行、全量收集所有相关数据。

```typescript
interface CollectedData {
  fg: {
    branch: FamilyGraphRoleBranch | null;
    treeText: string;
    rootProfile: PersonProfile | null;
    familyMembers: string[];
  };
  kb: Array<{ title: string; content: string }>;
  history: Array<{ role: string; content: string }>;
  portrait: string | null;
  context: {
    message: string;
    entities: string[];
    kinshipTerms: string[];
    pronounTarget: string | null;
  };
  knownFields: {
    hasAge: boolean;
    hasRelations: boolean;
    hasAppearance: boolean;
    hasOccupation: boolean;
    hasPersonality: boolean;
    askedPersonFound: boolean;
  };
}
```

### 第二步：ReadinessGate（就绪门）【核心新增】

**职责**：在 LLM 生成之前，检查数据是否足够。如果不够，**直接注入明确措辞**，不让 LLM 自己决定。

```typescript
interface ReadinessDecision {
  canAnswer: boolean;
  missingFields: string[];
  constraints: string[];
  antiFabricationGuard: string;
}
```

**决策逻辑**：

```
用户意图分类
  │
  ├─ 问人（谁/名字/多大/在哪）
  │   ├─ askedPersonFound=true  → 通过
  │   └─ askedPersonFound=false → "不清楚""没听说过"
  │
  ├─ 问年龄（多大了/几岁）
  │   ├─ hasAge=true  → 通过
  │   └─ hasAge=false → "你没告诉过我，我记不清了"
  │
  ├─ 问背景（说说你自己/你是怎样的人）
  │   ├─ hasPersonality=true  → 通过
  │   └─ hasPersonality=false → "我的事你知道的呀，你提醒我一下"
  │
  ├─ 问身份信息（名字/哪里人/做什么）
  │   ├─ hasIdentity=true  → 通过
  │   └─ hasIdentity=false → "这个你没跟我说过"
  │
  └─ 只聊天不涉及个人信息 → 通过
```

### 第三步：PromptAssembler（提示词装配器）

**职责**：按确定性顺序一次性装配完整的 `knowledgeBaseText`。**永不后覆盖前**。

```
Layer 1: 身份与规则
  buildRoleplayRules() → 规则①-⑥ + 角色身份覆盖

Layer 2: 硬性约束
  ReadinessGate.antiFabricationGuard → 如果数据不够

Layer 3: 画像与设定
  年龄锚点（从缓存画像提取，永不丢失）
  家族关系树（FG分支文本）
  知识库条目（KB搜索结果）
  角色画像（已知事实 + 未知边界）

Layer 4: 上下文
  历史扮演  角色风格指令
```

LLM 看到的最终结构：**规则 → 你不知道什么 → 你知道什么 → 上下文**

### 第四步：LLM Generate

复用现有 `ctx.m5.orchestrate()`，不做特殊处理。

### 第五步：Validator（验证器）

LLM 生成后，对照 DataCollector 的输出做交叉验证。

```typescript
function validate(reply, data): ValidationResult {
  // ① 身份保持检查：自称是否正确
  // ② 事实一致性检查：回复中的数字/日期是否在数据中
  // ③ 边界检查：是否违反未知边界
  // ④ 编造检查：人名是否在已知人物列表中
  
  switch (severity) {
    case 'error':   → 触发1次重生成，再失败用兜底话术替换
    case 'warning': → 打日志 + 探针标记
    case 'pass':    → 通过
  }
}
```

## 四、动态人设生长链路

针对没有预设档案的角色，设计完整的生长路径。

### 三阶生长规则

```
首次出现在对话中
    ↓
┌─ 第一阶：临时建档期（0~5轮互动） ──────────────┐
│  自动创建临时角色档案（存在角色域临时表中）          │
│  每轮提取：姓名/身份/年龄/性格/外貌/关系            │
│  信息只用于当前扮演，不写入主图谱                    │
└────────────────────────────────────────────────┘
    ↓ 达到阈值（≥5轮 + ≥8个信息点）
┌─ 第二阶：自动转正期 ────────────────────────────┐
│  角色信息同步写入 FamilyGraph 主表                │
│  标记 is_roleplay=1, source=roleplay             │
│  创建与用户的关系边，初始化亲密度/信任度            │
│  角色记忆从临时存储升级为长期三库流转               │
└────────────────────────────────────────────────┘
    ↓
┌─ 第三阶：持续丰满期 ────────────────────────────┐
│  每轮互动自动提取新信息，更新档案属性               │
│  和玉瑶本体的人物画像更新逻辑完全一致               │
│  所有信息带「来源对话」标记，可追溯                │
└────────────────────────────────────────────────┘
```

### 关键边界

- 所有自动提取的信息必须来自用户明确说过的内容，**禁止模型脑补**
- 档案所有字段带「信息来源」标记，可追溯到对应对话轮次
- 核心人设变更必须有用户明确指令

## 五、三库记忆体系对齐

角色记忆和玉瑶本体同标准、同算法、数据隔离。

### 三库对齐规则

| 层级 | 玉瑶本体 | 角色扮演 | 隔离机制 |
|:----:|---------|---------|---------|
| **砂金库** | conversations 表 | roleplay_id 标记 | 原始对话永久留存，压缩规则同主 |
| **金库** | 结构化语义记忆 | roleplay_id 专属 | 钙化/情感向量/实体标签复用主算法 |
| **黑钻** | 高情绪强度晋升 | 角色专属黑钻 | 高互动价值对话自动晋升 |

### 召回与隔离规则

| 场景 | 召回范围 |
|:----|---------|
| 激活角色扮演时 | 该角色的三库记忆 + 关系状态 + 人物档案 |
| 普通玉瑶对话 | **完全不触发**角色记忆 |
| 退出角色扮演 | 状态冻结存档，下次激活恢复 |

记忆衰减/巩固与主记忆共用一套算法。

## 六、全链路监管审查

```
事前约束层（ReadinessGate）
  → 数据够→安全回答
  → 数据模糊→弹性措辞
  → 无数据→直接说不知道

事中校验层（Validator）
  → 身份校验 → 自称/称呼是否符合人设
  → 事实校验 → 回复信息是否在已采集数据中
  → 边界校验 → 是否超出已知范围
  → 严重编造 → 1次重生成，再失败兜底替换

事后审计层（Auditor）
  → 完整链路留痕：输入→采集→判定→注入→回复→验证
  → 支持按角色/时间/问题类型回溯
  → 定期统计编造率/人设漂移率，异常自动告警
```

### 探针监控体系

| 探针组 | 指标 | 接入 |
|:------|------|:----:|
| 管线性能 | 采集耗时 / 装配耗时 / 生成耗时 | 现有 Hook 系统 |
| 业务质量 | 编造率 / 人设漂移度 / 记忆召回准确率 | 新增 H15-H17 |
| 运行状态 | 在线角色数 / 日互动轮次 / 转正角色数 | 新增 H18-H20 |

## 七、离线自学习与维护

### 四类离线任务

| 任务 | 频率 | 内容 | 守卫 |
|:----|:----:|------|:----:|
| 记忆巩固 | 每日凌晨 | 计算钙化分、执行晋升/衰减 | 不超出用户表达过的内容 |
| 档案补全 | 每日 | 从当日对话提取新信息补全档案 | 所有补全标注来源对话 |
| 关系更新 | 每日 | 更新亲密度/信任度，同步 FG | 只计算，不自动改核心人设 |
| 健康巡检 | 每周 | 检查档案完整性/编造率异常 | 异常自动标记待人工复核 |

### 守卫红线

- 所有离线修改不能超出用户明确表达过的内容范围
- 所有变更前有快照，支持一键回滚
- 离线任务不自动修改角色核心人设

## 十、验证场景

| 场景 | 输入 | 预期行为 | 验证方法 |
|:----|------|---------|---------|
| 有数据 | "扮演徐诗韵"→"说说你自己" | LLM 基于 FG/KB/画像真实数据回答 | 验证回复不包含不在数据中的信息 |
| 无年龄 | "你多大了" | ReadinessGate 注入"你没告诉过我" | 验证回复不含数字 |
| 不认识的人 | "我姐姐怎么样"(FG无数据) | 注入"我不清楚这个人" | 验证回复不含姓名/年龄/职业 |
| 无画像 | "说说你的事" | 注入"我的事你知道的呀"| 验证回复为推辞措辞 |
| 临时角色→转正 | 新角色聊 5 轮以上 | 自动创建 FG 条目 | 验证 FG 中新增节点 |
| 记忆隔离 | 切换角色 | 前一角色的记忆不泄漏 | 验证上下文中无前角色数据 |

## 十一、风险与回退

| 风险 | 概率 | 缓解 |
|:----|:----:|:------|
| 新管线影响正常对话 | 低 | 只在 `_currentRoleplay` 分支内生效 |
| DataCollector 过慢 | 低 | 并行采集 + 3s 超时 |
| ReadinessGate 过度约束 | 低 | 措辞保留弹性，"你提醒我一下" |
| 验证器误报 | 低 | 只打日志不阻塞回复 |
| 编造兜底失败 | 极低 | 再失败则用模板回复替换 |

## 十二、设计决策记录

| 决策 | 选择 | 理由 |
|:----|:----|:------|
| 角色域位置 | `src/app/roleplay/` | 与其它业务域平级，不侵入 engine 层 |
| 与主系统交互 | 接口调用，不直接操作 DB | 遵守主系统架构规则，不产生旁路 |
| 数据隔离 | 同库同表 + roleplay_id 标记 | 避免多库带来的同步/迁移问题 |
| 三阶生长阈值 | 5 轮 / 8 个信息点，可配置 | 适应不同角色成熟速度 |
| 离线任务 | 后台异步，不与实时管线耦合 | 不影响对话响应速度 |
| 重试策略 | 严重编造重生成 1 次，失败用兜底 | 平衡响应速度与回答质量 |
