# Demo 优化经验总结 → WenStarOS 平移指南

> 每次调优同步记录，踩坑经验全部留给新系统

---

## 一、架构踩坑记录

### 1.1 巨型单体文件（chat.ts 2980行）

**问题**：入口、检索、生成、持久化、角色扮演全部揉在一个文件，改一处牵全身。

**解决**：拆分为4条独立管线，单向依赖、无循环引用：
- `ChatEntryGuard.ts` — 入口守卫、扮演识别
- `ChatRetrievalPipeline.ts` — 记忆/知识库/FG检索
- `ChatGeneratePipeline.ts` — LLM调用、回复校验
- `ChatPersistencePipeline.ts` — 落库、副作用

**→ 新系统直接采用管线式架构，文件不超过500行**

### 1.2 角色路由三重重复分类

**问题**：`classify()` + `evaluateTransition()` 在 chat.ts / M5Orchestrator / DeepSeekLLMProvider 三处独立调用，三份状态变量，结果可能冲突。

**解决**：以 chat.ts 的 `_currentRole` 为唯一权威源，M5/DeepSeek 接收参数。新增 `ChatContext` 统一上下文。

**→ 新系统全局只有一个 `ChatContext`，所有模块只读访问**

### 1.3 同步持久化阻塞接口

**问题**：梦境队列、年轮里程碑、自我演化在主线同步执行，拖慢回复速度。

**解决**：全局 `AsyncQueue` 异步队列，非关键路径推入队列不阻塞。

**→ 新系统所有副作用走 EventBus 异步总线**

### 1.4 上下文块重复拼接

**问题**：`contextBlock` 拼接后又被 `knowledgeBaseText` 二次带入 `userMessage`，LLM 收到重复指令。

**解决**：分离上下文块与用户原始消息，userMessage 只传用户输入的原始文本。

**→ 新系统严格区分「上下文」「历史」「用户输入」三层**

## 二、参数调优记录

### 2.1 钙化（重要性）阈值

| 场景 | 参数 | 当前值 | 效果 |
|------|------|:------:|------|
| 金库晋升 | calcium_level ≥ 2 | 2 | 重要事件才会晋升 |
| 黑钻晋升 | calcium_level ≥ 2 + strength > 0.6 | 复合条件 | 极高门槛，只有真正重要的记忆进入 |
| 梦境触发 | calcium_level ≥ 2 | 2 | 强度足够才触发离线巩固 |
| 工作分类 | p.factual > 0.4 | 0.4 | 防止闲聊误判为工作 |

**经验**：钙化门槛不能过低，否则金库会被日常对话淹没。

### 2.2 向量相似度

| 参数 | 当前值 | 用途 |
|------|:------:|------|
| VECTOR_SIM_THRESHOLD | 0.5 | 黑钻向量相似度检索 |
| hybridSearch 语义权重 | 0.30 | ONNX 重排序权重 |
| hybridSearch 关键词权重 | 0.60 | keyword 检索权重 |

**经验**：语义分权重不宜超过关键词分——当前 TF-IDF 关键词效果优于 ONNX 小模型。

### 2.3 LLM 温度

| 场景 | 温度 | reasoning_effort | timeout |
|------|:----:|:----------------:|:-------:|
| 日常对话 | 0.9 | undefined | 10s |
| 回忆/分享 | 1.0 | undefined | 10s |
| 亲密场景 | 1.0 / 0.4(角色扮演) | max(角色扮演) | 20s |
| 角色扮演 | 0.4 | max | 15s |
| 简短模式 | 0.4 | undefined | 10s |

**经验**：角色扮演必须低温度+高推理，否则角色容易飘移。亲密场景需要高温度才有创造力。

### 2.4 角色切换熔断

| 参数 | 原值 | 新值 | 原因 |
|------|:----:|:----:|------|
| INTIMATE_THRESHOLD | 2 | 1 | 旧值需要连续2条才切恋人，第一条总被秘书压制 |
| MAX_SWITCHES | 3 | 8 | 旧值3次就触发熔断，亲密切换频繁被锁 |
| LOCK_DURATION | 5轮 | 3轮 | 旧值锁5轮太长，测试时几乎永久秘书 |
| DECAY_INTERVAL | 30min | 30min | 自然衰减节奏合理，保持 |

## 三、图谱实践经验

### 3.1 关系方向必须明确

**错误做法**：`sibling_of`（不区分长幼）→ LLM 猜错谁是姐谁是妹
**正确做法**：`elder_sister_of` / `younger_sister_of`（方向明确）

**→ 新系统所有关系类型必须分方向，禁止模糊关系**

### 3.2 "我"节点保护

**错误做法**：聊天空中"我"节点可能被误删，图谱基石断裂
**正确做法**：`_ensureSelfNode()` 启动时自动检查重建 + 自动备份

**→ 新系统 FG 必须有完整性自检 + 自动备份机制**

### 3.3 唯一数据库源

**错误做法**：两份 family_graph.db，默认路径与 server.ts 路径不同
**正确做法**：`DEFAULT_DB_PATH` 统一指向 `data/webui/knowledge/family_graph.db`

### 3.4 垃圾节点隔离

**问题**：对话误提取的 2 字词（"加班""舒服""阴蒂"等）被当成人名存入 FG
**解决**：定期巡检 + 人工复核 + **新增人名必须有 age/occupation/relation 至少一项**

**→ 新系统 FG 入库前必须经过人名校验过滤器**

## 四、亲密检测统一经验

### 4.1 三处正则不一致问题

之前 `INTIMATE_KEYWORDS` 在三个文件各自定义，正则不同，检测结果可能冲突。

**修复**：统一到 `src/common/utils/is-intimate.ts`，一处修改全局生效。

**→ 新系统所有"决策类"逻辑必须单源定义**

### 4.2 关键词策略

| 策略 | 内容 |
|------|------|
| 包含主动方 | `操你` `干你` |
| 包含被动方 | `操我` `摸我` |
| 包含身体部位 | `奶子` `胸` `屁股` |
| 包含常见误写 | `模`（模 vs 摸） |
| 排除学术词 | `研究` `学习` `人体解剖` 不触发亲密 |

### 4.3 亲密模式激活条件

```
level ≥ 2 → 直接激活 PassionateMode（20秒超时，宽松）
isDeepIntimate() → 消息含明确性词 → 激活
isMoan() → 纯语气词 → 激活
isAcademic() → 学术话题 → 跳过
```
